## 2. 背景与目标

### 2.1 业务背景

需要在仓库 `examples/finder` 中沉淀一个**小而完整**的查找能力示例，便于演示设计驱动流程与后续实现。典型场景包括：在文件内容中搜索关键字或模式，以及按名称/类型/深度等条件定位文件与目录。

### 2.2 目标

1. **文本查找**：在指定根目录下扫描文本类文件，按字面或正则匹配内容，输出命中位置（文件路径、行号、行内容片段）。
2. **文件/目录查找**：在指定根目录下按名称模式（**正则**，见 §4.1「名称模式」）、类型（文件/目录/二者）、可选最大深度等条件枚举路径。

### 2.3 非目标

- 全文索引、增量索引或后台守护进程式「监视」。
- 压缩包内搜索、Office/PDF 等非纯文本二进制解析。
- 跨机器分布式搜索或网络文件系统专用优化。
- GUI；本设计以 **CLI（命令行）** 为主，程序库 API 为可选扩展。

## 3. 范围

### 3.1 In Scope

- 单进程、本地文件系统上的递归遍历（可配置最大深度、是否跟随符号链接，默认**不跟随**以降低环与意外范围风险）。
- 文本查找：UTF-8 为主；对无法解码为文本的文件可 **跳过** 或 **计为跳过并输出诊断**（实现二选其一，在 README 与实现注释中固定说明）。
- 文件/目录查找：名称过滤、文件/目录筛选、深度限制；可选「仅统计文件数/目录数」**可放入后续迭代**，不作为 v1 验收阻塞项。
- **CLI 退出码**：采用 §8.4 约定表（**成功完成即为 0**，含零命中/零路径；用法错误 2；不可恢复错误 3。**不**采用类 grep 的「无匹配 → 1」）。
- `README.md` 中与上述能力一致的使用说明（若本仓库 `examples` 惯例需要）。

### 3.2 Out of Scope

- 编辑器集成、LSP、IDE 插件。
- 权限提升、绕过系统访问控制。
- 复杂查询语言（如 SQL）；仅保留 CLI 标志位与简单模式字符串。

## 4. 术语与约束

### 4.1 术语定义


| 术语    | 定义                                                                                                                         |
| ----- | -------------------------------------------------------------------------------------------------------------------------- |
| 根目录   | 用户指定的搜索起点路径；所有递归必须限制在其子树内（经规范化路径比较，禁止 `..` 逃逸）。                                                                            |
| 文本查找  | 以**行**为粒度在文件内容中匹配模式（`pattern`）。                                                                                            |
| 命中    | 一次成功的模式匹配，至少包含文件路径、行号、行文本（可截断）。                                                                                            |
| 名称模式  | `finder path --name` 的 `PATTERN` 为 **Python 正则**（`re` 模块）；对路径**最后一段**（`os.path.basename`）做 `re.search`；编译失败属不可恢复错误（退出码 3）。 |
| 深度    | 根目录深度为 0，每向子目录深入一层 +1。                                                                                                     |
| 规范化路径 | 解析符号链接策略由 `followSymlinks` 决定；比较「是否在根下」时，对路径做 OS 级规范化后再判断前缀/包含关系。                                                          |


### 4.2 全局约束

- **技术栈**：**Python 3.10+**；依赖与虚拟环境使用 **[uv](https://github.com/astral-sh/uv)** 管理（`pyproject.toml` / 锁文件按项目惯例）；依赖保持最少。
- **安全**：禁止将用户输入拼接进 shell 执行；路径 join 后必须仍在根目录子树内。
- **性能**：数千小文件量级下秒级可接受；避免明显 O(n²) 重复读盘。
- **合规**：不收集、不上传路径或文件内容；日志若含路径需在文档中提示敏感信息风险。

## 5. 业务流程

### 5.1 主流程

#### 5.1.1 文本查找（子命令 `finder text`）

1. 解析 CLI：`root`、`pattern`；`--regex`（默认，可省略）与 `--fixed-strings` 互斥，未指定时 **按正则** 解释 `pattern`；可选 `--max-depth`、`--glob` / `--glob-exclude`。
2. 校验 `root` 存在且为目录；否则走 §5.2。
3. 使用 Walker 遍历文件树（深度、glob 过滤、默认不跟随 symlink）。
4. 对候选文件按文本打开并逐行匹配（二进制启发式见 §9.2）。
5. 经 Output 模块将命中写入 stdout（格式见 §8.1）。
6. 按 §8.4 设置退出码。

#### 5.1.2 文件/目录查找（子命令 `finder path`）

1. 解析 CLI：`root`、`--name PATTERN`、`--type f|d|a`、`--max-depth`（可选）。
2. 校验 `root`；否则走 §5.2。
3. Walker 遍历并结合名称正则（§4.1）与类型过滤。
4. 路径每行一条输出至 stdout。
5. 按 §8.4 设置退出码（扫描**成功完成**即为 0，与是否列出路径无关）。

### 5.2 异常流程


| 场景            | 处理                                                                                       |
| ------------- | ---------------------------------------------------------------------------------------- |
| 根路径不存在或非目录    | stderr 说明；退出码 **3**（不可恢复错误）。                                                             |
| 模式非法（如正则编译失败） | stderr 说明；退出码 **3**。                                                                     |
| 单文件不可读        | 跳过该文件；可选 stderr 警告；整体继续；若扫描整体仍成功完成则退出码 **0**。                                            |
| 目录遍历权限不足      | 跳过该节点或子树；可选警告；不因此单独抬升为 3，除非实现无法继续。                                                       |
| 符号链接与环        | 默认不跟随 symlink；若未来支持跟随，须记录已访问 inode/真实路径防环。                                               |
| 用户中断（Ctrl+C）  | 优雅退出；**Unix / Unix 类系统**退出码 **130**（128+SIGINT 惯例）；**Windows** 行为以实现为准，须在 README 写明实际码值。 |

## 6. 模块设计

### 6.1 CLI 入口

- **职责**：解析参数与子命令、帮助信息、编排 Walker / ContentSearch / PathSearch / Output，映射 §8.4 退出码。
- **输入**：`argv`；环境变量（可选，如 `NO_COLOR`）。
- **输出**：进程退出码；stdout / stderr 文本。
- **依赖**：ContentSearch、PathSearch、Walker、公共配置类型、Output。
- **边界**：不包含目录遍历与模式匹配细节实现。
- **错误处理**：用法错误 → 退出码 2；不可恢复错误 → 3。
- **安全要求**：校验所有文件路径落在根目录子树内。

### 6.2 文件树遍历（Walker）

- **职责**：自顶向下枚举路径；应用 `maxDepth`、`followSymlinks`、类型过滤（文件/目录/全部）。
- **输入**：根路径、`SearchOptions`。
- **输出**：路径迭代器或异步/回调流（由实现选定）。
- **依赖**：标准库文件系统 API。
- **边界**：不做内容匹配、不做名称模式语义（名称过滤由 PathSearch 或上层完成，依实现拆分而定）。
- **错误处理**：权限错误：跳过并可选警告。
- **安全要求**：每次 join 后规范化并校验仍位于根下。

### 6.3 文本查找（ContentSearch）

- **职责**：对单个文件路径按行读取并匹配；输出 `TextHit` 列表。
- **输入**：文件路径、模式、是否正则、编码策略（UTF-8 为主）。
- **输出**：`TextHit` 列表。
- **依赖**：Walker 或上层提供的文件序列。
- **边界**：不解析 Office/PDF/压缩包等二进制格式。
- **错误处理**：编码失败按全局策略（跳过文件或计为跳过并诊断），在 README 固定一种。
- **安全要求**：单行输出长度上限（实现可配置常数），避免极端长行占满内存。

### 6.4 路径查找（PathSearch）

- **职责**：在 Walker 产出的路径上应用名称模式与类型过滤。
- **输入**：根路径、名称模式、`--type`、深度选项。
- **输出**：`PathHit` 或路径字符串列表。
- **依赖**：Walker。
- **边界**：不读取文件内容。
- **错误处理**：与 Walker 一致。
- **安全要求**：与 Walker 一致。

### 6.5 公共类型与格式化（Output）

- **职责**：将 `TextHit` / 纯路径格式化为 stdout 行（如 `path:line:content`）。
- **输入**：结构化命中或路径。
- **输出**：字符串行。
- **依赖**：无业务依赖（标准库即可）。
- **边界**：不含遍历与匹配逻辑。
- **错误处理**：无独立错误码；写入失败或编码问题由 CLI 捕获并映射为 §8.4。
- **安全要求**：可对控制字符做转义或剥离（可选实现）。

| 错误（stderr） | 人类可读消息                                                                                                                                  |

| 项          | 说明                                                                    |
| ---------- | --------------------------------------------------------------------- |
| 目的         | 按**名称与类型**查找路径                                                        |
| 形式         | CLI 子命令                                                               |
| 参数         | `<root>`；`--name PATTERN`（Python 正则，对 basename `re.search`）；`--type f |
| 响应（stdout） | 每行一条路径                                                                |
| 错误         | 同 §8.1                                                                |
| 幂等         | 是                                                                     |
| 权限         | 列目录与 stat                                                             |


### 8.3 命令：`finder --help` / `finder <subcommand> --help`

- 列出子命令与关键选项，与实现一致。

### 8.4 CLI 退出码约定


| 码   | 含义                                         | 适用                               |
| --- | ------------------------------------------ | -------------------------------- |
| 0   | **扫描成功完成**（含 **零条** stdout 输出：无命中或无匹配路径）   | 两子命令                             |
| 2   | 用法错误、未知参数、互斥选项冲突                           | 启动前失败                            |
| 3   | 不可恢复错误（根路径无效、`pattern` / `--name` 正则编译失败等） | 无法按用户意图完成扫描                      |
| 130 | 用户中断（SIGINT），**Unix / Unix 类系统**（128+2）    | 非 Windows；Windows 须在 README 单独说明 |


**说明**：与 `grep`（无匹配常返回 1）不同，本示例 **成功完成恒为 0**，便于脚本将「业务无结果」与「命令失败」区分时仅依赖 stderr/输出内容。

**SIGINT**：Unix 类系统固定 **130**；Windows 不以 130 为承诺时，README 须写实际退出码。

## 9. 状态与规则

### 9.1 状态机

无长期状态；单次调用为**无状态**只读扫描。

### 9.2 核心业务规则

1. 相对路径先相对当前工作目录解析，再与根目录约束结合；实现须在 README 写明解析顺序。
2. 默认 **不跟随** 符号链接。
3. 文本查找应对「明显二进制」做启发式跳过（例如读取首块检测 NUL），若未实现启发式则须在 README 说明「可能误读二进制」风险及建议扩展名过滤（可选）。
4. 退出码严格遵循 §8.4，与 §3.1、§5.2 一致。

## 10. 验收标准

### 10.1 功能验收

- `finder text`：在 fixture 下能命中指定正则；未传 `--fixed-strings` 时 **默认按正则**；`--fixed-strings` 下特殊字符按字面匹配。
- `finder path`：能列出匹配名称的文件、目录或二者（依 `--type`）。
- `--max-depth` 行为与 fixture 目录结构一致。
- 非法根路径、非法正则：退出码 **3**，stderr 有信息。
- 用法错误：退出码 **2**。
- 无命中/无路径：退出码 **0**（仍属成功完成）。
- `--help` 与实现一致。

### 10.2 质量验收

- 提供最小 fixture 目录或生成脚本，支撑自动化测试。
- 单元测试覆盖：路径规范化与「根下」校验、basename 正则（`--name`）、至少一种内容匹配。
- Windows 与 POSIX 路径行为差异在 README 中说明；若 v1 仅验证一方，须明确声明。

### 10.3 设计一致性要求

- 不得在未更新设计的情况下引入第三大类主功能（如网络搜索）。
- 安全规则（路径逃逸、symlink 默认策略）不得弱化；变更须升版并更新本规范化稿。

## 12. AI 实施附录

### 12.1 实现切片建议

1. **Slice A**：路径规范化、根下校验、Walker（深度、symlink 策略）。
2. **Slice B**：`finder path` + fixture + basename 正则（`--name`）。
3. **Slice C**：`finder text` + 编码与二进制启发式。
4. **Slice D**：README、§8.4 退出码、可 CI 运行的测试入口（若仓库已有 CI）。

### 12.2 推荐测试点

- 空目录；仅文件；仅子目录；混合深度。
- symlink 指向树外（不跟随时不得逃逸出根）。
- 含空格与 Unicode 的文件名。
- `--fixed-strings` 下正则特殊字符按字面匹配。

### 12.3 禁止事项

- 禁止硬编码某台机器上的绝对路径作为默认搜索根。
- 禁止静默忽略全部 IO 错误；须统计或警告。
