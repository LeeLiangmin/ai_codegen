# AI 设计驱动实现 Toolkit 总体方案

## 1. 目标

本方案用于在单服务、单仓库场景下，建立一套“根据设计文档驱动 AI 持续完成代码实现”的 toolkit / skill 体系。

本方案重点解决以下三个核心问题：

1. 代码生成过程的连续运行
2. 质量保证
3. 结果沉淀与过程控制

首阶段约束如下：

- 设计文档输入以 Markdown 为主
- PDF 可作为参考材料，但最终必须收敛为结构化 Markdown
- 首阶段聚焦单服务、单仓库
- 质量门禁以设计一致性检查与自动化测试为主，人工评审作为补充

---

## 2. 设计原则

### 2.1 文档先行

AI 不直接基于零散描述编码，而是基于结构化设计文档进行实现。

### 2.2 切片执行

实现过程以可独立验证的最小切片推进，避免一次性生成大批代码而失控。

### 2.3 状态可恢复

每个阶段都必须有显式状态记录，支持失败重试、断点续跑与人工接管。

### 2.4 验证内建

质量检查不是收尾动作，而是嵌入在每个切片完成后的必经关卡。

### 2.5 结果沉淀

所有关键决策、偏差、验证结果和交付索引都应形成可复用资产。

---

## 3. Toolkit 总体架构

Toolkit 分为六层：

### 3.1 文档规范层

负责定义设计文档格式，使设计既适合人工评审，也适合 AI 理解与执行。

输出包括：

- 设计文档模板
- 章节规范
- 接口描述规范
- 数据模型描述规范
- 验收标准描述规范

### 3.2 文档建模层

负责把设计文档解析为统一实现模型。

统一实现模型至少包括：

- 模块清单
- 接口清单
- 数据结构清单
- 约束清单
- 风险点清单
- 测试点清单
- 切片建议

### 3.3 编排执行层

负责把设计实现流程拆分为可连续运行的阶段任务。

关键能力包括：

- 阶段状态机
- 切片调度
- 失败分类
- 自动重试
- 断点恢复
- 人工接管入口

### 3.4 代码生成层

负责按切片完成实现。

输出对象包括：

- 业务代码
- 测试代码
- 配置文件
- 数据迁移脚本
- 辅助文档

### 3.5 质量保障层

负责对每个切片和整体结果进行自动验证。

关键检查包括：

- 设计一致性检查
- 格式检查
- 静态检查
- 类型检查
- 单元测试
- 契约测试
- 集成验证

### 3.6 沉淀控制层

负责将执行过程和结果转化为长期资产。

沉淀对象包括：

- 设计到实现的追踪矩阵
- 偏差说明
- 修复记录
- 运行报告
- 最终交付索引
- 可复用实现模板

## 4. Skill 体系设计

建议把 toolkit 组织为多个 skill，而不是单个大而全 skill。

### 4.1 `design-intake`

职责：

- 接收设计文档输入
- 检查文档是否满足结构化要求
- 识别缺失章节与不完整信息
- 输出 intake 报告

### 4.2 `design-normalize`

职责：

- 将补充说明、PDF 参考内容合并进统一设计稿
- 把设计收敛到结构化 Markdown
- 生成规范化设计文档

### 4.3 `design-to-plan`

职责：

- 将设计转为实现计划
- 提取模块、接口、数据结构、依赖关系
- 输出 backlog、优先级和验收点

### 4.4 `plan-to-slices`

职责：

- 将实现计划拆成最小可验证切片
- 为每个切片定义输入、产出、验证标准和依赖关系

### 4.5 `slice-implement`

职责：

- 按切片生成或修改代码
- 补齐必要测试与配置
- 控制修改范围与实现边界

### 4.6 `slice-verify`

职责：

- 检查代码与设计的一致性
- 执行自动化验证
- 生成切片验证报告

### 4.7 `integration-verify`

职责：

- 对多个切片组合结果执行集成验证
- 检查整体行为与端到端关键路径

### 4.8 `result-curate`

职责：

- 归档过程证据与结果报告
- 更新追踪矩阵与偏差记录
- 生成可继续迭代的上下文资产

---

## 5. 连续运行机制设计

### 5.1 状态机

建议采用如下执行阶段：

`Intake -> Normalize -> Plan -> Slice -> Implement -> Verify -> Curate -> Close`

### 5.2 运行单元

每次运行都应形成一个 run，对应一次完整的设计实现任务。

每个 run 下可包含多个 slice，每个 slice 都具备独立状态。

### 5.3 状态记录要求

每个阶段至少记录：

- 输入引用
- 执行动作
- 输出产物
- 校验结果
- 失败原因
- 下一步建议
- 人工干预点

### 5.4 失败分类

失败建议分为三类：

1. 文档问题：设计缺失、定义冲突、约束不明确
2. 实现问题：生成代码错误、依赖不满足、改动越界
3. 验证问题：测试失败、契约不满足、与设计不一致

### 5.5 恢复策略

- 同阶段可重试
- 切片失败不阻断其他无依赖切片规划
- 必要时可人工标记为 accepted deviation
- 所有恢复动作都必须记录原因和结果

---

## 6. 质量保证方案

### 6.1 一级：设计一致性检查

检查重点：

- 是否覆盖设计中定义的模块与接口
- 是否实现设计规定的关键约束
- 是否产生未授权功能扩展
- 是否违反命名、边界或依赖约束

### 6.2 二级：自动化技术检查

检查重点：

- 格式检查
- Lint / 静态分析
- 类型检查
- 单元测试
- 核心契约测试

### 6.3 三级：人工补充评审

适用情形：

- 自动修复反复失败
- 架构偏差较大
- 高风险模块改动
- 设计文档本身存在明显歧义

---

## 7. 结果沉淀与控制方案

### 7.1 必须沉淀的资产

每次 run 至少沉淀以下内容：

- 规范化设计输入
- 实现计划
- 切片列表
- 切片验证报告
- 设计实现追踪矩阵
- 偏差说明
- 最终交付索引

### 7.2 追踪矩阵

追踪矩阵建议建立以下映射：

- 设计章节 -> 模块
- 模块 -> 代码目录
- 接口 -> 实现文件
- 接口 -> 测试文件
- 需求点 -> 验收结果

### 7.3 偏差控制

任何未完全按设计实现的地方，都必须显式记录：

- 偏差项
- 偏差原因
- 替代方案
- 风险影响
- 后续处理建议

### 7.4 交付索引

最终要形成一份对 AI 和人工都友好的交付入口，用于继续迭代。

## 9. 首版落地建议

首版优先交付以下内容：

1. 设计文档规范模板
2. 设计转实现计划模板
3. 切片任务模板
4. 追踪矩阵模板
5. 运行状态文件格式
6. skill 说明文档

以上内容完成后，再逐步加入自动校验脚本和编排入口。
